### Week 6 Achievements

##### Core Database Concepts

A **Database** is a structured or semi-structured information system persisted on storage devices, designed to handle concurrent access from multiple applications simultaneously. Key components include:

| Concept | Description |
|---|---|
| **Session** | The lifecycle from establishing a connection to a database system through to disconnection. |
| **Primary Key** | A column (or combination of columns) that uniquely identifies every record in a table. |
| **Foreign Key** | A column that references the primary key of another table, enforcing referential integrity between relations. |
| **Index** | A data structure that accelerates data retrieval at the cost of additional storage and slower write operations. |
| **Partition** | Splitting a large table into smaller, more manageable segments to improve query performance on subsets of data. |
| **Execution Plan** | The step-by-step query execution strategy generated by the query optimizer to select the most efficient path. |
| **DB Log** | A write-ahead record of all modifications, used for crash recovery and replication synchronization between primary and replica nodes. |
| **Buffer** | A temporary in-memory area that holds recently read or written data blocks before they are flushed to disk, reducing I/O latency. |

##### RDBMS & NoSQL

**RDBMS (Relational Database Management System):**

- Organizes data into tables with rows and columns; relationships between entities are expressed through primary and foreign keys.
- Uses **SQL** as the standard language for querying and data manipulation.
- Guarantees **ACID** properties — Atomicity, Consistency, Isolation, and Durability — making it suitable for transaction-intensive workloads.

**NoSQL (Not Only SQL):**

- Departs from the rigid table-based model in favor of flexible, schema-optional data structures.
- Common data model variants:

| Type | Examples |
|---|---|
| Document-based | MongoDB |
| Key-Value | Redis, DynamoDB |
| Wide-column | Cassandra |
| Graph | Neo4j |

- Best suited for large-scale, unstructured, or rapidly evolving datasets that require elastic horizontal scaling.

##### OLTP vs. OLAP

| Characteristic | OLTP | OLAP |
|---|---|---|
| **Purpose** | Process real-time transactional operations | Analyze aggregated historical data |
| **Data profile** | Frequently updated, row-oriented | Read-heavy, aggregated, column-oriented |
| **Typical applications** | Banking, e-commerce, ticketing systems | Business intelligence, reporting, data analysis |
| **Optimization focus** | Transaction throughput and concurrency | Complex query response time |
| **AWS services** | RDS, Aurora | Redshift, Athena, QuickSight |

##### Amazon RDS (Relational Database Service)

A fully managed service that abstracts the operational overhead of provisioning, patching, and managing relational databases:

- **Supported engines:** Aurora, MySQL, PostgreSQL, MariaDB, Oracle, and SQL Server.
- **Key capabilities:**
  - **Automated backups** — continuous DB and log backups with a configurable retention window of up to 35 days.
  - **Read Replicas** — offload read-heavy queries (e.g., reporting) from the primary instance.
  - **Multi-AZ deployment** — synchronous standby replica with automatic failover in the event of a primary failure.
  - **Encryption** — data protected at rest (AES-256) and in transit (TLS).
  - **Auto Scaling** — dynamically adjusts both storage capacity and instance size to match demand.
  - **Network security** — access controlled through Security Groups and Network ACLs.
- Primarily targeted at **OLTP** workloads requiring strong consistency and transactional guarantees.

##### Amazon Aurora

A cloud-native relational database engine purpose-built for high performance and availability, fully compatible with both MySQL and PostgreSQL:

- Inherits all core RDS capabilities, and further extends them with:
  - **Backtrack** — rewind the database to a precise earlier point in time without restoring from a snapshot.
  - **Clone** — creates an independent copy of the database almost instantly with minimal additional storage cost.
  - **Global Database** — spans multiple AWS regions with a single primary writer and up to 16 read-only replicas per region.
  - **Multi-Master** — enables concurrent write operations across multiple nodes, eliminating single-writer bottlenecks.
- Storage is distributed across **three Availability Zones** in six-way replication, ensuring durability and seamless automatic recovery.

##### Amazon Redshift

AWS's fully managed **Data Warehouse** service, built on a columnar adaptation of PostgreSQL optimized for OLAP-scale analytics:

- Employs a **Massively Parallel Processing (MPP)** architecture:
  - **Leader Node** — parses and orchestrates query execution across compute nodes.
  - **Compute Nodes** — store data in columnar format and execute parallel query fragments.
- **Columnar storage** means only the columns referenced in a query are read from disk, dramatically reducing I/O for analytical queries.
- Additional capabilities include:
  - Native **SQL, JDBC, and ODBC** connectivity.
  - **Redshift Spectrum** — run SQL queries directly against data files stored in Amazon S3 without loading them.
  - **Pause/Resume** (transient cluster mode) — stop the cluster when idle to avoid unnecessary compute charges.

##### Amazon ElastiCache

A fully managed, in-memory caching service that dramatically reduces the load on underlying databases and improves response times for frequently accessed data:

- Supports two caching engines: **Redis** and **Memcached**.
- AWS manages node provisioning, patching, failure detection, and automatic node replacement.
- **Redis** is generally the preferred choice due to its richer feature set:
  - Replication and high-availability clustering.
  - Data persistence across restarts.
  - Pub/Sub messaging and Lua scripting support.
- Application-level cache invalidation logic must be carefully designed to prevent stale data from being served after the underlying database is updated.
